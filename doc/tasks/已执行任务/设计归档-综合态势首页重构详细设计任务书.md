# 综合态势首页重构详细设计任务书

## 任务主题

参考 `https://www.crucix.live/` 的页面组织方式，结合当前项目实际代码结构，重新设计并改造“综合态势首页”。

---

## 一、任务背景与本次改造的根本目标

### 1. 当前问题不是“有没有地图”，而是“地图没有成为真正的主体”

当前项目首页已经有地图、点位、图层、筛选、右侧情报流、自动轮播逻辑，但这些内容的组织方式仍然偏“后台页面”，没有形成真正的“态势驾驶舱”。

当前主要问题如下：

1. 地图虽然已经在首页中部，但周边信息组织方式不对；
2. 顶部与地图内部信息层级仍不够清晰；
3. 用户想要的“轮播框”没有以正确形态出现在页面中；
4. 当前自动轮播被做成了地图右侧的“重点情报流”，不是地图下方的“态势轮播承接层”；
5. 地图缩放和边界限制过强，导致放大后边缘区域看不到；
6. 页面没有真正学到参考站那种“地图是主战场，其他信息围绕地图服务”的组织方式。

### 2. 本轮改造目标

本轮不是做普通 UI 微调，而是要把首页改造成：

**“以地图为绝对核心，左侧负责总览与控制，右侧负责焦点解释，底部负责动态轮播承接”的综合态势驾驶舱首页。**

### 3. 本轮参考对象的核心学习点

本轮必须明确学习 `CRUCIX` 的不是“全球 3D 地球效果”，而是它的页面组织逻辑：

1. 顶部只保留轻量状态条；
2. 中央主视觉必须极强；
3. 侧边区域只承担解释、说明、附加信息；
4. 持续变化的信息流应作为主视觉承接层，而不是被塞成普通侧栏；
5. 页面第一眼必须先看核心态势，再看细节。 ([Crucix][2])

## 二、必须先承认的当前项目现状

### 1. 当前项目技术栈与约束

当前仓库是 Vue 3 + TypeScript + Vite，依赖里有 `leaflet`、`echarts`、`element-plus`、`pinia`，说明当前首页地图是基于 Leaflet 体系做的，不是高德/百度原生地图实现。 ([GitHub][3])

### 2. 当前首页相关核心文件

本轮改造必须围绕以下文件进行，不允许脱离现有代码结构空想：

* `src/components/dashboard/HaidianCockpitPanel.vue`
* `src/components/dashboard/HaidianCockpitMapStage.vue`
* `src/mock/dashboardCockpit.ts`
* 视情况新增首页专用子组件文件

### 3. 当前页面里已经存在的有效基础

当前代码并不是从零开始，已经有以下可复用能力：

1. 已有地图主组件与地图舞台组件拆分；
2. 已有自动轮播逻辑；
3. 已有轮播与地图点位联动逻辑；
4. 已有图层筛选；
5. 已有专题联动；
6. 已有基于图层的说明文案。 ([GitHub][4])

### 4. 当前页面最关键的误区

当前误区不是“没做轮播”，而是：

* 已有 `currentTicker` 自动轮播；
* 已有 `activeTickerIndex`；
* 已有切换后 `focusPoint(nextItem.relatedPointId)`；
* 但它被渲染成了右侧的“重点情报流”。 ([GitHub][4])

这导致用户一直觉得“轮播框没做出来”，因为当前交付的视觉形态不是用户要的那种“地图底部承接式横向轮播框”。

### 5. 当前地图边界问题已经能在代码中定位

当前地图组件里直接设置了：

* `maxBounds: L.latLngBounds(props.mapBounds)`
* `maxBoundsViscosity: 0.85`
* 初始化 `fitBounds(L.latLngBounds(props.mapBounds).pad(-0.08))`

这说明当前地图被强约束在一个很紧的海淀边界框内，所以用户放大后会感觉“边缘看不到”“视野被裁死”。 ([GitHub][5])

## 三、本轮对“轮播框”的重新定义

### 1. 轮播框的正式定义

本项目中的“轮播框”，定义如下：

**轮播框 = 位于主地图下方、横向展开、可自动播放、可手动切换、与地图和右侧详情实时联动的动态事件承接区。**

### 2. 它不是什么

它不是以下任意一种东西：

* 不是右侧竖向事件列表；
* 不是顶部状态条；
* 不是普通的事件表格；
* 不是单纯的时间筛选器；
* 不是地图上的弹窗集合；
* 不是右侧“重点情报流”的另一种叫法。

### 3. 它的职责

它必须承担以下职责：

1. 作为地图主视觉的下方承接层；
2. 持续滚动显示当前图层下最重要的事件；
3. 提供用户对当前态势的“节奏感”；
4. 驱动地图点位高亮与切换；
5. 驱动右侧焦点详情同步变化；
6. 当用户点击地图点位时，反向同步到轮播框当前项。

## 四、首页重构后的完整布局设计

---

## 1. 顶部区域设计

### 1.1 顶部定位

顶部必须改为**薄状态条**，高度轻量，不能再占据首屏核心空间。

### 1.2 顶部放置内容

顶部仅允许放以下内容：

#### 左侧

* 系统名称：海淀区综合态势驾驶舱
* 当前区域标识：海淀区

#### 中间

* 当前图层名称
* 当前底图状态
* 数据更新时间
* 在线状态 / 运行状态

#### 右侧

* 专注视图
* 重置视图
* 全屏（如当前项目已支持可保留，否则先预留按钮位）

### 1.3 顶部禁止事项

禁止继续在顶部放置：

* 大标题块
* 大面积统计卡片横条
* 冗长说明文案
* 多层说明信息堆叠
* 会把地图压下去的视觉模块

### 1.4 顶部设计目标

用户进入页面后，顶部只能起到“快速定场”和“最轻量控制”的作用，不能抢走地图主视觉。

---

## 2. 左侧区域设计

### 2.1 左侧定位

左侧区域是“态势总览 + 快速控制区”，它不是详细解释区，也不是轮播区。

### 2.2 左侧整体分为两层

#### 第一层：态势摘要区

固定显示四个最重要的摘要卡片：

1. 高风险点位
2. 突发事件
3. 热点片区
4. 重点关注

这四项内容应作为“总览摘要”固定露出，不要藏入二级面板。当前 mock 数据里已经围绕风险预警、突发事件、热点事件、重点关注等图层做了配置说明，本轮应将这些概念继续保留并强化为左侧总览。 ([GitHub][6])

#### 第二层：快速控制区

固定露出最关键的控制项：

* 图层切换
* 时间范围
* 风险等级
* 片区筛选
* 观察方式/显示模式

### 2.3 左侧显示原则

1. 最常用筛选项必须首屏可见；
2. 次级设置可折叠；
3. 不要让用户先点一个“设置”按钮，才看到全部控制能力；
4. 左侧宽度适中，不能抢地图空间；
5. 左侧内容必须以“支持地图阅读”为目标，而不是形成独立信息墙。

### 2.4 左侧卡片细化建议

#### 摘要卡片字段建议

每张摘要卡片建议包含：

* 主数值
* 单位/标签
* 环比或趋势小标识
* 一句辅助说明
* 点击后对应切换地图焦点或图层

#### 快速控制区字段建议

每个控制项建议包含：

* 控件名称
* 当前值
* 是否激活
* 变更后联动的地图行为说明

---

## 3. 中央地图区域设计

### 3.1 地图定位

地图必须成为页面绝对主角，是首页第一视觉中心。

### 3.2 地图面积要求

首页主体区应优先把最大可用面积给地图，不允许被顶部、右侧、左侧反向挤压。

### 3.3 地图视觉表达要求

必须做到：

1. 海淀区边界高亮；
2. 海淀外围区域弱化；
3. 点位清晰分层；
4. 缩放后层级感明显；
5. 高优先级点位有明显视觉突出；
6. 低优先级点位不造成地图信息污染。

### 3.4 地图信息层建议

中央地图区应包含以下几层：

#### 第一层：底图层

本轮建议按“高德优先、百度预留、OSM 退居备用”的方向执行，但不要在本轮文档里要求一次性把所有 provider 都完全做完；优先先把地图承载关系设计好。

#### 第二层：区域边界层

* 海淀区主边界
* 重点片区/街道/网格预留层
* 可支持后续更细行政层级扩展

#### 第三层：事件点位层

按类型区分：

* 风险预警
* 突发事件
* 热点事件
* 重点关注

#### 第四层：高亮联动层

* 当前选中点位高亮
* 当前轮播项点位高亮
* 当前筛选结果重点高亮

#### 第五层：说明辅助层

* 少量重点事件允许使用引导线/说明框
* 大量事件时禁止全部拉线，避免地图变乱

### 3.5 地图交互要求

地图点击行为必须分两类：

#### 点击点位

* 地图高亮该点
* 右侧详情切换到该点
* 底部轮播框切换到对应事件
* 如当前轮播项不是该事件，则同步切换当前播放项

#### 点击区域

* 支持切换片区/区域焦点
* 左侧筛选与右侧详情可同步更新
* 但不应强制覆盖轮播框逻辑，除非该区域明确绑定一批事件

---

## 4. 右侧区域设计

### 4.1 右侧定位

右侧区域是“焦点解释区”，只能做解释、详情和联动入口，不能再承担主轮播职责。

### 4.2 右侧分三段

#### 第一段：当前焦点详情

显示当前点位/当前事件/当前片区的核心说明：

* 标题
* 事件类型
* 风险等级
* 时间
* 区域
* 状态
* 摘要
* 查看详情按钮

当前 ticker 数据结构里本身就有 `title/time/area/level/status/summary/relatedPointId`，这部分可以直接作为底部轮播项与右侧详情的共同基础字段。 ([GitHub][6])

#### 第二段：专题联动

保留当前已有的“专题联动”能力，但位置改成右侧中段更合理，因为它本质上是“对当前焦点的扩展检索入口”，不是地图主交互入口。 ([GitHub][4])

#### 第三段：持续观察列表

这里可以放小型列表，例如：

* 最近更新
* 关联对象
* 连续监测对象
* 本区域最近 3 条变动

### 4.3 右侧禁止事项

* 不要再把右侧竖向滚动列表当作主轮播框；
* 不要让右侧内容比地图更抢眼；
* 不要堆太多统计图表；
* 不要做成“第二个首页”。

---

## 5. 地图下方轮播框设计（本次核心）

### 5.1 轮播框定位

轮播框必须紧贴主地图底部，作为地图的“态势承接层”。

### 5.2 轮播框视觉要求

必须是明显的横向区域，不能是一个小条，不能是一个不起眼的滚动字。

### 5.3 轮播框组成结构

建议分为四个部分：

#### A. 左侧控制区

包含：

* 自动播放/暂停
* 上一条
* 下一条
* 当前播放状态
* 当前图层标识

#### B. 中间主内容区

显示当前轮播项的核心信息：

* 时间
* 标题
* 类型标签
* 区域
* 等级
* 状态
* 一句摘要

#### C. 右侧快捷动作区

包含：

* 定位地图
* 查看详情
* 锁定当前项
* 进入列表

#### D. 底部或次级列表区

显示若干条可切换项缩略列表：

* 当前项高亮
* 其他项横向排列
* 点击即可切换

### 5.4 轮播框必须支持的行为

#### 自动轮播

使用当前已有 ticker 自动轮播逻辑作为基础，不要重写整套业务。当前代码里已经按 5200ms 的间隔推进 `activeTickerIndex` 并调用 `focusPoint`。 ([GitHub][4])

#### 手动切换

必须支持：

* 上一条
* 下一条
* 直接点击指定条目

#### 联动地图

每次轮播切换时：

* 地图自动高亮对应点位
* 必要时平滑移动到对应点位
* 打开该点的视觉焦点状态

#### 联动右侧详情

每次轮播切换时：

* 右侧详情区切换为当前轮播项内容
* 专题联动同步跟随当前焦点

#### 图层切换联动

切换图层后：

* 轮播列表自动切换为当前图层对应数据
* 当前播放项重置到新图层第一项
* 地图同步刷新高亮对象

### 5.5 轮播框字段定义建议

建议每条轮播项标准结构如下：

* `id`
* `title`
* `time`
* `area`
* `layer`
* `level`
* `status`
* `summary`
* `relatedPointId`
* `detailTarget`（若后续详情页需要）
* `priority`（如后续需要排序）

当前 mock 中 ticker 已经具备 `id/layer/title/time/area/level/status/summary/relatedPointId`，本轮应尽量复用。 ([GitHub][6])

## 六、交互链路详细设计

### 1. 页面初始进入

1. 页面加载
2. 默认聚焦海淀区域
3. 默认展示当前主图层
4. 左侧摘要和控制区可见
5. 右侧展示当前第一焦点
6. 底部轮播框自动开始播放
7. 地图同步高亮第一条轮播对应点位

### 2. 用户点击左侧图层按钮

1. `activeLayer` 切换
2. 当前可见点位切换
3. 轮播框数据切换到该图层
4. 当前播放项重置为第一项
5. 地图高亮对应第一项点位
6. 右侧详情同步更新

### 3. 用户点击地图点位

1. 地图记录选中点位
2. 底部轮播框定位到对应事件项
3. 当前播放项切换
4. 右侧详情切换为该事件
5. 自动播放进入新的当前位置继续运行或保持暂停状态

### 4. 用户点击底部轮播项

1. 当前轮播项切换
2. 地图聚焦到对应点位
3. 右侧详情同步变化
4. 若当前处于自动播放模式，则从新位置继续轮播

### 5. 用户点击右侧专题联动

1. 跳转或打开搜索入口
2. 但不应破坏当前地图焦点状态
3. 返回后仍保留当前选中项

## 七、地图边界与视野设计要求

### 1. 根本要求

必须把“默认聚焦海淀”和“强制把地图锁死在海淀边界内”区分开。

### 2. 当前必须修改的问题

当前地图不应继续使用“强粘性边界 + 紧边 fit”作为常态浏览体验，因为这会导致：

* 放大后边缘不可看；
* 用户视野被裁；
* 地图像被装进小框里；
* 真实在线地图感变弱。 ([GitHub][5])

### 3. 修改原则

1. 默认进入时可聚焦海淀；
2. 用户缩放后，应允许看到更自然的边缘和周边空间；
3. 增加“聚焦海淀”按钮，而不是把聚焦逻辑写死在所有场景里；
4. 点位飞行聚焦后，不应马上又被强制 fit 回原边界；
5. 只有在用户明确执行“重置视图/聚焦海淀”时，才重新回到默认视野。

## 八、视觉层级与排版原则

### 1. 第一眼优先级

页面第一眼必须是：

1. 中央地图
2. 地图下方轮播框
3. 左侧态势摘要
4. 右侧焦点详情
5. 顶部薄状态条

### 2. 绝对不能再出现的情况

* 大标题压住地图
* 大卡片压住地图
* 轮播框不明显
* 右侧列表比底部轮播更抢眼
* 地图区域像中间插图而不是主战场

### 3. 视觉节奏

页面节奏必须是：

**先看地图 —— 再看底部正在播什么 —— 再看右侧解释 —— 最后用左侧控制深入筛选**

而不是：
**先看一堆统计卡片 —— 再找地图 —— 再找事件 —— 再猜哪里是轮播**

## 十、验收标准（必须逐条满足）

### 1. 布局验收

* 首页第一眼必须先看到地图主体；
* 顶部变薄；
* 左侧为摘要与快速控制；
* 右侧为焦点解释；
* 底部有明显横向轮播框。

### 2. 轮播框验收

* 轮播框不是右侧竖向列表；
* 轮播框支持自动播放；
* 轮播框支持暂停/继续；
* 轮播框支持上一条/下一条；
* 点击轮播项可以联动地图和右侧详情；
* 切换图层后轮播内容同步变化。

### 3. 地图验收

* 地图仍然是主视觉；
* 点位点击联动正常；
* 放大后边缘可见；
* 不再被海淀边界强裁切；
* 可通过按钮回到默认聚焦视图。

### 4. 结构验收

* 不推翻整个项目结构；
* 改动集中在首页驾驶舱相关组件；
* 尽量复用现有 ticker、point、topic 数据结构；
* 文档同步更新完整。

1. 禁止继续把右侧“重点情报流”解释成用户要的轮播框；
2. 禁止继续做“大标题 + 大卡片 + 地图在下面”的后台式首页；
3. 禁止一次性重写整个综合态势页面；
4. 禁止丢弃现有 ticker 逻辑从零造轮子；
5. 禁止在一个步骤里同时做布局、逻辑、地图 provider、样式、文档全部重写；
6. 禁止忽略参考站“地图主导 + 下方持续信息流”的核心组织方式；
7. 禁止把底部轮播框做成不起眼的小条或隐藏模块。
