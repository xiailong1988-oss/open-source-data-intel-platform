# 基于参照网站的首页版式映射设计文档

## 适用对象

* 当前项目：`open-source-data-intel-platform`
* 当前页面：综合态势首页 / 海淀区驾驶舱首页
* 参照站：`https://www.crucix.live/`


## 一、本文档的唯一目的

本文档不是讨论“驾驶舱应该怎么设计”，也不是让 Codex 自己发挥。

本文档只做一件事：

**把参照网站的页面骨架，逐位置映射到你当前项目首页。**

也就是说，先确定：

* 参照站这个位置是什么
* 你项目这个位置必须放什么
* 允许怎么改
* 不允许怎么变形

本轮执行时，Codex 必须先遵守本文档中的**位置映射关系**，再处理代码实现细节。
未经允许，不得自行更改主布局骨架。

## 二、先明确：参照网站到底参考什么，不参考什么

### 1. 本轮参考的是“版式骨架”

本轮参考 `CRUCIX`，参考的是以下内容：

1. 页面主视觉关系；
2. 地图/地球在页面中的视觉权重；
3. 顶部信息条的轻量化处理；
4. 地图旁边持续动态信息区的位置；
5. 解释性信息与主体信息的主从关系；
6. 页面第一眼先看哪里、第二眼看哪里、第三眼看哪里。

### 2. 本轮不参考的内容

以下内容不是本轮重点，不要本末倒置：

1. 不强求复刻 3D 地球；
2. 不强求完全一样的科技风装饰；
3. 不强求黑色主题完全一致；
4. 不强求一比一还原动画；
5. 不强求把你项目做成全球 OSINT 站点。

### 3. 一句话总结

本轮参考的是：

**“页面结构”**

不是：

**“页面皮肤”**

## 三、参照网站的版式骨架拆解

根据参照站当前可见页面，可以确认的版式骨架如下：

### 1. 顶部极轻

页面顶部只承担“状态感”和“系统激活感”，不是大块功能区。可见文案包括 `CRUCIX`、`TERMINAL ACTIVE`，整体是非常轻的顶部提示，不抢主体地图位置。([Crucix][1])

### 2. 中央主体绝对主导

页面中央主视觉是地图/地球，且主视觉压强极强。页面的交互提示如 `SCROLL TO ZOOM · DRAG TO PAN`、`GLOBE MODE` 都围绕主体地图而不是独立工具栏展开。([Crucix][1])

### 3. 侧边解释层独立存在

参照站右侧能看到 `Signal Guide` 及其解释内容，说明侧边区域承担的是“解释层/说明层”的职责，而不是去抢占主体。([Crucix][1])

### 4. 页面不是“后台卡片首页”

参照站当前可见页面不是“顶部一堆统计卡片 + 中间一块地图 + 底部再拼模块”的后台式结构，而是**先看主体，再看围绕主体的信息层**。([Crucix][1])

## 四、你当前项目首页的真实现状

在开始改之前，必须先承认现状，不能假装现在是从零开始。

### 1. 当前页面已经有地图主体

你当前首页确实已经是围绕地图区展开，不是没有地图主位。这个方向没错。([GitHub][2])

### 2. 当前地图上方有两层覆盖

当前 `HaidianCockpitPanel.vue` 里，地图上方同时存在：

* `haidian-cockpit__signal-strip`
* `haidian-cockpit__toolbar`

这两块都绝对定位在地图上层，而且 `signal-strip` 是四列 grid，`toolbar` 在右上角单独一组按钮。现在的问题不是“没有态势摘要和图层控制”，而是**做成了两层且偏厚**。([GitHub][2])

### 3. 当前右侧已经有情报流

当前页面右侧已经有 `haidian-cockpit__intel-rail`，文案明确写着“重点情报流”“沿地图右侧持续滚动，点击即可反向聚焦地图点位”。也就是说，**重要情报流放右侧**这个方向，你当前代码其实已经做对一半了。([GitHub][2])

### 4. 当前 ticker 自动轮播已经存在

当前自动轮播逻辑会按 5200ms 推进 `activeTickerIndex`，并对下一条执行 `focusPoint(nextItem.relatedPointId)`。说明“轮播/联动”本身不是没有，而是展示位置和点击行为不对。([GitHub][2])

### 5. 当前点击情报项不会打开详情

当前 `selectTickerItem(index)` 只做三件事：

* 找到当前条目
* 设置 `activeTickerIndex`
* 执行 `focusPoint(target.relatedPointId)`

它不会直接打开详情页。这个必须改。([GitHub][2])

### 6. 当前 ticker 数据量太少

当前 mock 里 `ticker` 只有少量条目，能看到至少从 `TK-HD-01` 到 `TK-HD-03` 起步，而你之前测试时也已经感觉到量感不足。当前 `currentTicker` 自动轮播逻辑如果数据量不够，就很难形成真正的“滚动流”。([GitHub][3])

### 7. 当前地图视野仍然偏死

地图初始化时用了 `maxBounds` 和 `maxBoundsViscosity: 0.85`，并且 `fitBounds(...pad(-0.08))` 不止初始化时用，后续重置时也会再次用到，所以用户放大后觉得边缘看不到，是有代码依据的。([GitHub][4])

## 五、最终版式映射总图

这次不再让 Codex 自己想结构，直接按下图执行：

```text
页面最上层：极轻状态条
┌────────────────────────────────────────────────────────────┐
│ 系统名 | 当前区域 | 当前时间 | 运行状态 | 轻操作按钮         │
└────────────────────────────────────────────────────────────┘

主体区：参照 CRUCIX 的“主体优先 + 侧边解释层”
┌───────────────────────────────────────────────┬─────────────────────┐
│                 中央主地图工作区                │   右侧重要情报滚动流   │
│                                               │                     │
│  地图顶部窄横条：态势摘要 + 图层轻控制          │ 持续滚动、多条数据、     │
│                                               │ 当前项高亮、可点击详情  │
│  海淀主地图 + 点位 + 边界 + 联动                │                     │
│                                               │                     │
└───────────────────────────────────────────────┴─────────────────────┘
```

### 关键说明

1. 这次**不再做底部主轮播框**；
2. 这次**不再做左侧大面积摘要卡区**；
3. 这次首页重点是：

  * 地图顶部一条窄横条
  * 地图右侧重要情报滚动流
  * 地图主体本身最大化

## 六、逐区域版式映射

---

## A区：参照站顶部轻量信息区 → 你项目的页面最上层状态条

### 参照站此处特征

* 极轻
* 只起定场作用
* 不抢主视觉
* 不承担主体业务展示 ([Crucix][1])

### 你项目这里应该放什么

只放：

1. 系统名称：海淀区综合态势驾驶舱
2. 当前区域：海淀区
3. 当前时间 / 最新更新时间
4. 运行状态 / 在线状态
5. 专注视图 / 重置视图 / 全屏预留按钮

### 这里绝对不能放什么

1. 大标题块
2. 大统计卡片横条
3. 复杂筛选面板
4. 长段说明文案
5. 任何会把地图挤下去的东西

### 执行要求

如果当前页面最上层还有过厚的标题和卡片，必须压缩到极轻状态条。

---

## B区：参照站中央主视觉 → 你项目的海淀区主地图

### 参照站此处特征

* 这是页面绝对主角
* 所有解释层围绕它服务
* 交互提示围绕它展开 ([Crucix][1])

### 你项目这里应该放什么

1. 海淀区主地图
2. 海淀区主边界高亮
3. 外围区域弱化
4. 点位分层展示：

  * 风险预警
  * 突发事件
  * 热点事件
  * 重点关注
5. 地图点击联动
6. 当前情报流高亮项对应点位联动

### 这里绝对不能变成什么

1. 一块被四周模块挤压得很小的地图
2. 一块中间插图式地图
3. 背景装饰板
4. 只有看看而没有交互的静态图

### 执行要求

地图必须继续占据页面最大面积。
凡是会压缩地图面积的模块，都优先向“轻量化、窄条化、侧栏化”收缩。

---

## C区：参照站主体上方轻量覆盖提示 → 你项目地图顶部窄横条

### 参照站此处特征

参照站在主体上方有轻量的激活状态和模式提示，但不是厚工具栏。([Crucix][1])

### 你项目这里应该放什么

地图顶部只保留**一条窄横条**，并且只容纳两类信息：

#### 1. 态势摘要

只保留最关键的 4 项：

* 高风险点位
* 突发事件
* 热点事件
* 重点关注

#### 2. 图层轻控制

只保留图层切换按钮：

* 风险预警
* 突发事件
* 热点事件
* 重点关注

如确有需要，可保留极少量附加控制：

* 时间
* 区域
  但必须是次级或折叠，不得塞满窄条。

### 这里必须怎么显示

1. 窄
2. 横向
3. 轻
4. 一眼扫过即可
5. 摘要高亮，但不是大卡片

### 这里绝对不能再怎么做

当前页面已经把这里拆成：

* `signal-strip`
* `toolbar`

而且 `signal-strip` 还是四列 grid 卡片样式，`toolbar` 又是另一层按钮组。这个方向必须收回，改成**一条真正的窄横条**。([GitHub][2])

### 执行要求

1. 合并当前两层上方覆盖块；
2. 不再分裂成两块视觉区域；
3. 不再做厚卡片；
4. 不再做大工具栏；
5. 不再做传统后台式筛选面板直接露出。

---

## D区：参照站侧边解释层 → 你项目的右侧重要情报滚动流

### 参照站此处特征

参照站右侧有 `Signal Guide`，说明侧边区域承担的是解释层、连续信息层，而不是主视觉本身。([Crucix][1])

### 你项目这里应该放什么

这里不再放普通“右侧说明面板”，而是固定放：

**重要情报滚动流**

### 为什么一定放右侧

1. 你已经明确要求：重要情报滚动流应在地图右侧或左侧，不在底部；
2. 当前代码本来就已经有右侧 `intel-rail`，这说明从实现上延续右侧最省改动、最贴近现状；
3. 参照站本身也有明显的侧边解释层位置。([GitHub][2])

### 右侧重要情报滚动流必须放什么

每条至少包含：

1. 时间
2. 标题
3. 类型
4. 区域
5. 状态 / 风险等级
6. 一句短摘要

### 右侧重要情报滚动流必须有什么效果

1. 持续滚动感
2. 当前项高亮
3. 内容量充足
4. 能点击
5. 点击后打开详情页
6. 地图同步聚焦
7. 当前图层切换后，情报流内容同步切换

### 这里绝对不能退化成什么

1. 右侧静态普通列表
2. 只有四五条数据的弱列表
3. 点击后只聚焦地图、不打开详情的半成品
4. 单纯的“说明文字侧栏”

### 执行要求

右侧 `intel-rail` 不删除，但要升级成真正的“滚动流”，不是简单保留现状。([GitHub][2])

---

## E区：参照站点击后进入解释和深挖 → 你项目的详情打开逻辑

### 参照站此处特征

参照站页面的解释层是在帮助用户继续理解，而不是止步于“看一眼”。([Crucix][1])

### 你项目这里必须怎么对应

当前点击 ticker 条目不能只做：

* `activeTickerIndex = index`
* `focusPoint(...)`

还必须：

* 打开对应详情页 / 业务详情页 / 智能检索结果页

当前代码里 `selectTickerItem` 只聚焦点位，必须改。([GitHub][2])

### 推荐实现方式

优先复用现有 point / topic 的详情跳转链路，不要重新造一套路由体系。
如果点位数据里已有详情目标，则 ticker 点击时：

1. 定位 ticker 对应 point
2. 复用 point 的详情链接
3. 地图联动 + 路由跳转同时完成

## 十、最终验收标准

### 1. 版式验收

* 页面最上层是极轻状态条；
* 地图是绝对主视觉；
* 地图顶部只有一条窄横条；
* 重要情报滚动流在地图右侧；
* 不存在底部主轮播框。

### 2. 顶部窄横条验收

* 态势摘要和图层控制都在同一条窄横条里；
* 不是两块；
* 不是厚卡片；
* 不是后台工具栏。

### 3. 右侧滚动流验收

* 明显是“流”，不是普通列表；
* 数据量足够；
* 当前项高亮明显；
* 图层切换后同步变更；
* 点击后能打开详情页。

### 4. 地图验收

* 地图面积最大；
* 边界清晰；
* 点位联动正常；
* 当前情报项能联动地图；
* 放大后不再明显被裁边。

### 5. 代码结构验收

* 改动聚焦在首页相关组件；
* 尽量复用现有 ticker 和 point 逻辑；
* 不搞全站大重构；
* 文档与代码同步。
