# 任务：GIS态势图与交互问题整改

## 任务背景
当前项目已经完成一轮工作台化界面重构，但综合态势页仍然偏向“模块堆叠式首页”，GIS 地图尚未成为真正的信息组织核心。与此同时，系统内仍需从真实使用视角再做一轮显性点击、跳转、联动、弹窗、抽屉和菜单交互排查，确保整个平台在演示状态下稳定可用。

## 本次任务目标
1. 将综合态势页重构为以北京 GIS 态势图为核心的情报工作台。
2. 让风险预警、重点事件、热点区域、专题切换围绕 GIS 主视图组织，而不是平铺并列。
3. 对主要页面进行一次显性交互巡检，修复菜单、按钮、卡片、抽屉、弹窗和联动类问题。
4. 保持既有路由、业务钻取、mock 数据驱动和工作台风格体系稳定。

## 设计与实现原则
1. 不做普通后台式美化，必须真正重构综合态势页的信息主次。
2. 北京 GIS 底图必须成为首页第一视觉中心，并承担核心联动入口职责。
3. 图层切换、热点区域、预警点位、重点事件和专题入口必须围绕地图组织。
4. 页面显性交互问题必须通过实际构建与回归验证，不允许仅凭静态阅读代码判断。
5. 新增地图结构、交互逻辑和关键筛选/跳转代码必须补充清晰中文注释。

## 预期产出
1. 综合态势页完成 GIS 核心化升级，具备北京底图、专题图层、区域联动和情报侧栏。
2. 首页形成“关键态势总览 -> GIS 主视图 -> 联动情报面板 -> 趋势与处置追踪”的清晰层级。
3. 主要页面显性交互 BUG 完成一轮排查和修复，并在任务记录中写清楚修复结果。
4. 文档状态、任务记录和交互规范同步更新，便于后续继续开发。

## 验收方式
1. `npm run build` 通过。
2. `npm run smoke:ui` 通过，且重点链路不出现菜单失效、按钮失效或抽屉遮罩残留问题。
3. 综合态势页在视觉上以 GIS 为绝对核心，并能完成至少以下联动：图层切换、区域点击、点位点击、右侧情报面板联动、跳转检索/预警/事件/报告。
4. 当前任务文档、当前执行状态和交互规范在任务结束后同步更新。

## 本次改动说明

### GIS 态势图改造内容
1. 接入北京 16 区行政区 GeoJSON 底图，替换原有抽象片区示意图。
2. 将首页地图语义改为北京真实区县体系，并重做热点区域、风险点位、重点事件、专题研判 mock 数据。
3. 在 GIS 主视图区内新增图层切换、时间/区域/风险/对象筛选、热点区域观察、重点事件流、专题切换建议。
4. 重做右侧情报面板，使其承接区域/点位选中态、关联预警、事件、报告和实体跳转。

### 页面结构调整内容
1. 首页第一屏改为“简短工作台标题 + 顶部态势指标 + GIS 主舞台”。
2. 将快捷入口和态势摘要移至 GIS 主舞台之后，避免在第一视图与地图争抢权重。
3. 将辅助分析图谱和态势追踪看板下沉为第二层内容，形成更清晰的主次关系。
4. 保留原有统计卡片、图表钻取、列表跳转和报告/预警/任务联动能力。

### 交互测试范围
1. 首页 GIS 图层切换、热点区域选择、重点事件点击、右侧情报面板联动。
2. 智能检索结果详情抽屉打开/关闭及回到列表后的继续交互。
3. 智能问答首次进入、历史问题切换、推荐问题继续追问。
4. 预警记录详情抽屉打开/关闭、标记已处理与列表继续点击。
5. 综合态势 -> 智能检索 -> 综合态势 -> 数据接入 -> 智能问答 -> 报告中心 的菜单链路。

### 修复与收口的显性交互问题
1. 修复首页地图不具备真实 GIS 承载能力的问题，改为北京底图驱动。
2. 修复综合态势页信息结构过于平均的问题，地图不再是附属模块。
3. 补充地图加载失败重试、空点位空态、图例显隐和视图重置反馈。
4. 补充 UI 冒烟脚本中的抽屉、推荐问题、地图联动等关键链路验证。
5. 本轮回归中未发现新的阻断性菜单失效、抽屉残留遮罩或按钮无反馈问题。
