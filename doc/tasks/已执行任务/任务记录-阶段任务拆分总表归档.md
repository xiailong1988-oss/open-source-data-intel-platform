# Codex执行任务拆分文档

## 1. 文档信息

- 文档名称：Codex执行任务拆分文档
- 文档用途：将《情报平台界面与产品结构改造需求文档》拆解为可逐步执行的开发任务
- 使用对象：Codex、开发人员、项目负责人
- 执行方式：按阶段推进，逐任务完成，逐项验收
- 核心要求：**每完成一个任务，都必须保证项目可运行、文档可同步、代码可理解、结果可验收**

---

## 2. 总体执行原则

## 2.1 执行目标

将当前系统逐步改造为：

**以地图为入口、以事件和告警为主线、以专题工作台为视角、以智能问答为主入口、以简报输出为结果形式的情报分析工作平台。**

## 2.2 Codex执行硬规则

1. 不允许只改颜色和局部样式，必须先改产品结构
2. 不允许一次性大范围推翻现有代码，必须分阶段可运行改造
3. 不允许新页面和旧页面风格割裂，必须统一主题和组件
4. 不允许只做静态页面，必须预留或接入真实接口
5. 不允许首页仍然只是菜单入口页，必须改造成工作台
6. 不允许所有逻辑都堆在单个页面文件中，必须组件化
7. 不允许每个页面各自写一套通用逻辑，必须抽公共模块
8. 不允许遗漏空态、加载态、异常态、刷新状态
9. 不允许新增函数没有注释，后续新增代码必须补清晰注释
10. 不允许只改前端不更新文档，所有结构性变更必须同步 `doc` 目录

## 2.3 单任务交付要求

每个任务执行后必须同时产出：

1. 改动说明
2. 涉及文件清单
3. 新增组件说明
4. 涉及接口说明
5. 风险点说明
6. 验收结果说明
7. 文档更新记录

---

## 3. 推荐执行顺序

本项目建议按以下顺序执行：

1. 基础重构阶段
2. 首页工作台阶段
3. 事件中心阶段
4. 告警中心阶段
5. 专题工作台阶段
6. 智能检索与问答阶段
7. 简报中心阶段
8. 信源与数据健康阶段
9. 地图联动增强阶段
10. 收尾优化与统一验收阶段

---

## 4. 阶段拆分总览

### Phase 1：基础重构
目标：建立统一布局、统一主题、统一组件、统一状态处理机制

### Phase 2：首页综合态势工作台
目标：把首页从普通后台改造成情报总览工作台

### Phase 3：事件中心
目标：形成“列表 + 时间线 + 地图 + 详情”的事件主线页面

### Phase 4：告警中心
目标：形成“分级 + 证据链 + 可解释”的告警研判页面

### Phase 5：专题工作台
目标：先落地经济专题与创新专题两个工作台

### Phase 6：智能检索与问答
目标：把智能入口前置并接入上下文问答

### Phase 7：简报中心
目标：形成正式汇总输出链路

### Phase 8：信源与数据健康
目标：增强系统可信度、可维护性和可解释性

### Phase 9：地图联动增强
目标：强化地图作为系统主入口和联动核心的能力

### Phase 10：统一收尾
目标：统一样式、补齐注释、补齐文档、补齐验收
