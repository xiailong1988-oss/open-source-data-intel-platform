# 12-基础登录、权限、日志完善

## 一、任务目标
Task-10 只围绕系统运行所需的基础认证、当前用户信息、轻量角色控制和操作日志能力落地，不扩展复杂 RBAC、菜单权限树、单点登录或第三方统一认证。

本轮目标：
1. 提供登录、退出登录、当前用户接口。
2. 为后台管理接口增加基础保护。
3. 为关键管理操作增加自动操作日志。
4. 提供默认管理员初始化方案。

---

## 二、登录与鉴权实现说明
### 2.1 认证方案
当前采用轻量内存 token 会话方案：
1. 用户调用 `POST /api/auth/login`，使用用户名和密码登录。
2. 登录成功后，后端生成随机 token，并在内存中保存会话。
3. 后续请求通过 `Authorization: Bearer <token>` 或 `X-Auth-Token: <token>` 传递登录态。
4. 应用重启后，内存 token 会失效，需要重新登录。

说明：
1. 当前方案简单、可控，适合单体项目早期联调。
2. 当前未引入 Redis、JWT 黑名单或复杂会话集群能力。

### 2.2 密码处理
1. 当前密码采用工程内统一轻量哈希规则存储。
2. 默认管理员密码不会以明文写入数据库。

### 2.3 当前用户上下文
1. `AuthInterceptor` 会解析 token。
2. 解析成功后将当前用户放入 `UserContextHolder`。
3. 业务层和自动填充、操作日志可直接复用当前用户上下文。

## 三、接口清单
1. `POST /api/auth/login`
2. `POST /api/auth/logout`
3. `GET /api/auth/current-user`

### 3.1 登录接口 `POST /api/auth/login`
请求字段：
1. `username`
2. `password`

返回字段：
1. `accessToken`
2. `tokenType`
3. `expiresAt`
4. `currentUser`

`currentUser` 字段：
1. `userId`
2. `username`
3. `realName`
4. `phone`
5. `email`
6. `roleCodes`
7. `roleNames`
8. `expiresAt`

### 3.2 退出接口 `POST /api/auth/logout`
说明：
1. 需要携带有效 token。
2. 后端会移除当前 token 会话。

### 3.3 当前用户接口 `GET /api/auth/current-user`
说明：
1. 需要携带有效 token。
2. 用于前端回显当前登录用户及角色信息。

## 四、默认初始化账号说明
通过 Flyway 脚本 `V9__init_auth_admin_seed.sql` 初始化：

默认角色：
1. `ADMIN`
2. `ANALYST`

默认管理员：
1. 用户名：`admin`
2. 初始密码：`Admin@123456`
3. 默认角色：`ADMIN`

说明：
1. 若数据库中已经存在同名角色或账号，脚本不会重复插入。
2. 当前管理员账号主要服务于后台联调和基础验收。

## 五、管理类接口保护方式说明
### 5.1 角色注解
当前新增 `@RequireRole` 注解，用于对管理接口做基础角色保护。

当前已保护接口范围：
1. 数据源管理接口 `CollectionSourceController`
2. 采集任务管理接口 `CollectionTaskController`
3. 数据包管理接口 `DataPackageController`
4. 交付记录管理接口 `DeliveryRecordController`
5. 情报管理中的后台能力接口：
   - `GET /api/intel/page`
   - `GET /api/intel/{id}`
   - `POST /api/intel`
   - `PUT /api/intel/{id}`
   - `DELETE /api/intel/{id}`

当前策略：
1. 具备 `ADMIN` 角色的用户可访问上述管理接口。
2. 首页联调接口如 `ticker`、`stream`、`dashboard`、`map`、`regions` 保持公开访问。

### 5.2 登录态注解
当前新增 `@LoginRequired` 注解，用于保护需要登录但不一定要求特定角色的接口：
1. `POST /api/auth/logout`
2. `GET /api/auth/current-user`

## 六、操作日志实现说明
### 6.1 记录方式
当前使用 AOP 切面自动记录操作日志，不将日志逻辑写入各业务 Service。

自动记录范围：
1. 所有 `@RequireRole` 保护的接口
2. `AuthController` 下的认证接口

### 6.2 写入内容
自动写入 `sys_oper_log`：
1. `module_name`
2. `action_name`
3. `operator_id`
4. `operator_name`
5. `request_uri`
6. `request_method`
7. `request_param`
8. `response_data`
9. `success_flag`
10. `operate_time`

说明：
1. 模块名优先取 Controller 上的 `@Tag`。
2. 操作名优先取接口方法上的 `@Operation`。
3. 请求参数和返回结果会做基础序列化和长度截断，避免日志过大。
4. `password`、`accessToken`、`token`、`authorization` 等敏感字段会自动脱敏后再落库。

## 七、验收步骤
### 7.1 数据库验收
执行 SQL：

```sql
SELECT installed_rank, version, description, success
FROM flyway_schema_history
ORDER BY installed_rank;

SELECT username, real_name, status
FROM sys_user
WHERE username = 'admin';

SELECT role_code, role_name
FROM sys_role
WHERE role_code IN ('ADMIN', 'ANALYST');
```

预期结果：
1. Flyway 已执行到 `V9`
2. 存在默认管理员 `admin`
3. 存在默认角色 `ADMIN`、`ANALYST`

### 7.2 接口验收
1. 调用 `POST /api/auth/login`，使用 `admin / Admin@123456`
2. 从响应体获取 `accessToken`
3. 携带 `Authorization: Bearer <token>` 调用 `GET /api/auth/current-user`
4. 携带 token 调用 `GET /api/sources/page`
5. 调用 `POST /api/auth/logout`
6. 再次调用 `GET /api/auth/current-user`，应返回未登录错误

### 7.3 保护效果验收
1. 不携带 token 调用 `GET /api/sources/page`，应返回未登录
2. 不携带 token 调用 `POST /api/packages`，应返回未登录
3. 不携带 token 调用 `GET /api/intel/ticker`，仍应可以访问

### 7.4 操作日志验收
执行一次登录、查询管理接口或新增管理数据后，检查：

```sql
SELECT module_name, action_name, operator_name, request_uri, success_flag, operate_time
FROM sys_oper_log
ORDER BY id DESC
LIMIT 20;
```

预期结果：
1. 认证操作和后台管理操作会写入日志
2. 成功与失败状态可区分

## 八、已知限制与待后续补充项
1. 当前 token 会话存储在内存中，暂不支持多节点共享。
2. 当前只做基础角色控制，不提供复杂 RBAC 和菜单权限树。
3. 当前未实现修改密码、刷新 token、用户管理、角色管理页面。
4. 当前未接入单点登录、LDAP、OAuth2 或第三方统一认证。
