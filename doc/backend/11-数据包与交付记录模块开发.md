# 11-数据包与交付记录模块开发

## 一、任务目标
本轮严格围绕 Task-09 落地数据包与交付记录模块，只支撑当前前端联调与后台基础管理，不扩展复杂文件中心、对象存储集成、断点续传或审批流。

本次交付内容：
1. 数据包分页、详情、新增、编辑、发布。
2. 数据包文件列表查看。
3. 交付记录分页、新增、详情。
4. 预留文件下载与文件存储扩展字段。

---

## 二、本次新增表结构
### 2.1 `biz_data_package`
用途：
1. 维护数据包基本信息。
2. 维护当前发布状态与发布版本。
3. 汇总包内文件数量和总大小。

核心字段：
1. `package_name`：数据包名称。
2. `package_type`：数据包类型编码。
3. `stat_month`：统计月份，格式 `yyyy-MM`。
4. `region_scope`：地区范围。
5. `file_count`：文件数量。
6. `total_size`：总大小。
7. `status`：状态，当前支持 `draft`、`published`。
8. `publish_version`：发布版本号。
9. `publish_time`：最近发布时间。
10. `published_by_name`：最近发布人。
11. `publish_note`：最近发布说明。

说明：
1. 为满足“草稿/已发布”管理与前端联调，本次将状态字段设计为字符串状态码。
2. 当前只保留最近一次发布快照，不单独落地发布历史表。

### 2.2 `biz_data_package_file`
用途：
1. 维护数据包文件清单。
2. 支撑详情页文件列表展示。
3. 为后续文件中心预留存储和下载字段。

核心字段：
1. `package_id`
2. `file_name`
3. `file_path`
4. `file_size`
5. `file_type`
6. `checksum`
7. `storage_type`
8. `storage_bucket`
9. `storage_object_key`
10. `download_url`
11. `sort_order`

### 2.3 `biz_delivery_record`
用途：
1. 维护交付记录。
2. 记录接收方、交付方式、交付时间。
3. 快照数据包名称和版本，便于历史追溯。

核心字段：
1. `delivery_type`
2. `related_id`
3. `related_name`
4. `related_version`
5. `receiver_name`
6. `receiver_org`
7. `contact_phone`
8. `delivery_time`
9. `delivery_method`
10. `description`

## 四、接口字段说明

2. `pageSize`
3. `keyword`
4. `packageType`
5. `statMonth`

返回字段：
1. `id`
2. `packageName`
3. `packageType`

5. `statMonth`
6. `regionScope`
7. `fileCount`
8. `totalSize`

10. `statusLabel`
11. `publishVersion`
12. `publishTime`
13. `publishedByName`
14. `description`
15. `createdTime`

### 4.2 数据包详情 `GET /api/packages/{id}`
详情额外返回：
1. `publishNote`
2. `remark`
3. `updatedTime`

### 4.3 数据包新增/编辑
提交字段：
1. `packageName`
2. `packageType`
3. `statMonth`
4. `regionScope`
5. `description`
6. `remark`
7. `files`

`files` 子项字段：
1. `fileName`
2. `filePath`
3. `fileSize`
4. `fileType`
5. `checksum`
6. `storageType`
7. `storageBucket`
8. `storageObjectKey`
9. `downloadUrl`
10. `sortOrder`
11. `remark`

说明：
1. 编辑时采用“重建当前文件清单”的最小策略。
2. 系统会自动更新 `fileCount` 和 `totalSize`。

### 4.4 数据包发布 `POST /api/packages/{id}/publish`
可选提交字段：
1. `publishTime`
2. `publishedByName`
3. `publishNote`

发布规则：
1. 数据包至少存在一条文件清单后才允许发布。
2. 每次发布会将 `publishVersion` 自增。
3. 当前阶段仅维护最近一次发布快照。

3. 新增时会快照 `relatedName` 与 `relatedVersion`。

## 五、数据包与交付模块联调说明
### 5.1 页面联调顺序
建议按以下顺序联调：
1. 先调 `GET /api/packages/page` 打通列表页。
2. 再调 `POST /api/packages` 和 `PUT /api/packages/{id}` 打通表单页。
3. 使用 `GET /api/packages/{id}` 与 `GET /api/packages/{id}/files` 完成详情和文件清单页。
4. 使用 `POST /api/packages/{id}/publish` 打通发布动作。
5. 最后调 `GET /api/delivery-records/page`、`POST /api/delivery-records`、`GET /api/delivery-records/{id}` 完成交付记录页。

### 5.2 前端需要关注的约束
1. 数据包表单中的 `statMonth` 必须按 `yyyy-MM` 提交。
2. 文件清单至少维护一条后才允许发布。
3. 交付记录新增前，应确保所选数据包已经发布。
4. 当前文件下载只是字段预留，`downloadUrl` 可为空，不代表后端已提供下载接口。

### 5.3 返回结构约定
1. 所有接口统一返回 `ApiResponse<T>`。
2. 分页接口统一返回 `ApiResponse<PageResponse<T>>`。
3. 分页结果字段固定为 `records`、`total`、`pageNum`、`pageSize`。

## 六、启动方式
1. 启动 MySQL，并创建 `haidian_intel_platform` 数据库。
2. 进入 `backend` 目录。
3. 执行 `mvn spring-boot:run`。
4. 应用启动后 Flyway 会自动执行到 `V8`。

## 七、验收方式
### 7.1 数据库验收
执行 SQL：

```sql
SHOW TABLES LIKE 'biz_data_package';
SHOW TABLES LIKE 'biz_data_package_file';
SHOW TABLES LIKE 'biz_delivery_record';

SELECT installed_rank, version, description, success
FROM flyway_schema_history
ORDER BY installed_rank;
```

预期结果：
1. `flyway_schema_history` 包含 `V1` 到 `V8`。
2. 三张 Task-09 业务表创建成功。

### 7.2 接口验收
建议顺序：
1. 调用 `POST /api/packages` 新增一个带文件清单的数据包。
2. 调用 `GET /api/packages/page` 验证列表字段与分页结构。
3. 调用 `GET /api/packages/{id}` 与 `GET /api/packages/{id}/files` 验证详情和文件清单。
4. 调用 `POST /api/packages/{id}/publish` 验证状态、版本、发布时间回填。
5. 调用 `POST /api/delivery-records` 验证交付记录新增。
6. 调用 `GET /api/delivery-records/page` 和 `GET /api/delivery-records/{id}` 验证历史快照信息。

## 八、已知限制与待后续补充项
1. 当前未实现真正文件上传、下载、对象存储集成。
2. 当前未实现独立发布历史表，只保留最近一次发布快照和版本号。
3. 当前交付记录仅支持数据包交付，未扩展到专题服务、光盘等更多交付类型。
4. 当前编辑数据包时采用文件清单整体重建策略，后续如需细粒度编辑可再补充。
