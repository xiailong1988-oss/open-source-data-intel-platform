## 四、Task-08 新增表结构说明
### 4.1 `biz_collection_source`
用途：
1. 支撑“数据源管理”页面的分页、详情、新增、编辑、删除
2. 维护数据源分类、接入方式、启用状态和运行状态
3. 为采集任务提供可关联的数据源主表

关键字段：
- `source_name`：数据源名称
- `source_category`：数据源分类编码
- `source_url`：接入地址
- `access_type`：接入方式编码
- `enabled`：启用状态，`1` 启用、`0` 停用
- `status`：运行状态编码，当前约定 `running/stopped/abnormal`
- `region_scope`：地区范围说明
- `owner_name`：负责人
- `latest_collect_time`：最近采集时间
- `description`：描述

索引说明：
1. `uk_collection_source_name`：约束数据源名称唯一
2. `idx_collection_source_category`：支撑分类筛选
3. `idx_collection_source_access_type`：支撑接入方式筛选
4. `idx_collection_source_enabled`：支撑启停筛选
5. `idx_collection_source_status`：支撑运行状态筛选

### 4.2 `biz_collection_task`
用途：
1. 支撑“采集任务管理”页面的分页、详情、新增、编辑、删除
2. 维护采集任务启停状态和运行状态
3. 为后续真实采集调度接入预留最小结构

关键字段：
- `task_name`：任务名称
- `source_id`：关联数据源 ID
- `task_type`：任务类型编码
- `cron_expr`：调度表达式或调度规则文本
- `frequency_desc`：前端直接展示的执行频率
- `enabled`：启用状态，`1` 启用、`0` 停用
- `run_status`：运行状态编码，当前约定 `running/paused/abnormal`
- `latest_result`：最近执行结果编码，当前约定 `success/failed/partial_success`
- `latest_run_time`：最近执行时间
- `record_count`：累计采集记录数
- `success_rate`：成功率
- `description`：描述

索引与约束说明：
1. `uk_collection_task_name`：约束任务名称唯一
2. `idx_collection_task_source_id`：支撑按数据源筛选
3. `idx_collection_task_type`：支撑按任务类型筛选
4. `idx_collection_task_enabled`：支撑启停筛选
5. `idx_collection_task_run_status`：支撑运行状态筛选
6. `fk_collection_task_source_id`：维护任务与数据源的一对多关系

## 五、统一字段约定
当前业务表统一遵循以下规则：
1. 主键统一使用 `BIGINT`
2. 字符集统一使用 `utf8mb4`
3. 逻辑删除字段统一使用 `deleted`
4. 审计字段统一使用 `created_by / created_time / updated_by / updated_time`
5. 备注字段统一使用 `remark`

---

## 六、初始化方式
### 6.1 创建数据库
```sql
CREATE DATABASE IF NOT EXISTS haidian_intel_platform
DEFAULT CHARACTER SET utf8mb4
DEFAULT COLLATE utf8mb4_general_ci;
```

### 6.2 启动应用触发迁移
```bash
cd backend
mvn spring-boot:run
```

应用启动后会自动执行 `V1` 到 `V10`。

说明：
1. `V6` 提供首页热点与重点关注演示数据
2. `V7` 创建数据源与采集任务管理表
3. `V8` 创建数据包与交付管理表，不实现复杂文件中心
4. `V9` 会初始化默认角色和默认管理员账号，便于当前阶段快速联调
5. `V10` 会补充首页和后台管理联调演示数据，无需额外脚本系统

## 八、Task-09 新增表结构说明
### 8.1 `biz_data_package`
用途：
1. 支撑数据包管理页的分页、详情、新增、编辑与发布。
2. 维护数据包版本、发布时间、发布人、文件数量与总大小。
3. 为交付记录提供可追溯的数据包主表。

关键字段：
1. `package_name`：数据包名称。
2. `package_type`：数据包类型编码。
3. `stat_month`：统计月份，当前约定格式为 `yyyy-MM`。
4. `region_scope`：地区范围描述。
5. `file_count`：当前包内有效文件数。
6. `total_size`：当前包内有效文件总大小。
7. `status`：状态码，当前阶段采用 `draft/published`。
8. `publish_version`：发布版本号，自增维护。
9. `publish_time`：最近一次发布时间。
10. `published_by_name`：最近一次发布人名称。
11. `publish_note`：最近一次发布说明。

说明：
1. 为了让前后端更清晰地区分“草稿”和“已发布”，本次将 `status` 设计为 `VARCHAR(32)` 状态码，而不是早期草案里的 `TINYINT`。
2. 当前只保留最近一次发布快照和发布版本号，不单独新增发布历史表。

### 8.2 `biz_data_package_file`
用途：
1. 支撑数据包文件清单查看。
2. 在数据包新增、编辑时维护最小文件清单能力。
3. 预留后续文件存储与文件下载扩展字段。

关键字段：
1. `package_id`：所属数据包 ID。
2. `file_name`：文件名称。
3. `file_path`：逻辑文件路径。
4. `file_size`：文件大小。
5. `file_type`：文件类型。
6. `checksum`：校验值。
7. `storage_type`：预留存储类型。
8. `storage_bucket`：预留存储桶。
9. `storage_object_key`：预留对象存储键。
10. `download_url`：预留下载地址。
11. `sort_order`：展示顺序。

### 8.3 `biz_delivery_record`
用途：
1. 支撑交付记录分页、新增和详情。
2. 记录交付对象、交付方式、接收信息与交付时间。
3. 对数据包名称和发布版本做快照，避免后续名称调整影响历史记录。

关键字段：
1. `delivery_type`：交付类型编码，当前任务仅支持 `data_package`。
2. `related_id`：关联业务 ID，当前指向数据包 ID。
3. `related_name`：关联业务名称快照。
4. `related_version`：交付时的数据包发布版本快照。
5. `receiver_name`：接收人。
6. `receiver_org`：接收单位。
7. `contact_phone`：联系电话。
8. `delivery_time`：交付时间。
9. `delivery_method`：交付方式编码。
10. `description`：交付说明。

## 十、Task-11 初始化演示数据说明
`V10__init_demo_data_for_integration.sql` 会补充以下联调用初始化数据：

### 10.1 地区、事件类型与地图基础检查结果
1. 复用 `V2` 已初始化的海淀区、重点街道镇和四类地图图层。
2. 补充 `青龙桥街道`、`万寿路街道` 两个地区及边界占位数据，扩展地区切换场景。
3. 补充 `网络舆情`、`安保巡检`、`活动保障`、`重点人员关注` 四种事件类型，覆盖更多情报分类。
4. 地图图层本次不新增记录，原因是 `risk/emergency/hotspot/focus` 四类图层已经可以承接新增事件类型展示。

### 10.2 情报与首页演示数据
1. 初始化 10 条情报数据，其中 8 条可用于首页公开接口，2 条用于后台状态联调。
2. 情报数据覆盖不同地区、不同事件类型、不同发布时间，并包含地图坐标。
3. 数据时间采用相对 `NOW()` 方式生成，避免首页快讯条因时间过旧而为空。
4. 补充首页可直接展示的热点专题、重点关注对象，并关联到具体情报详情 ID。

### 10.3 后台管理演示数据
1. 初始化 5 条数据源演示数据，覆盖运行中、异常、停用状态。
2. 初始化 5 条采集任务演示数据，覆盖运行中、异常、暂停和不同执行结果。
3. 初始化 4 个数据包、8 个文件清单、3 条交付记录，覆盖草稿与已发布状态。

### 10.4 适合前端联调的查询口径
1. 首页公共接口可直接使用初始化后的公开情报、热点、重点关注和地图点位数据。
2. 后台管理接口可使用默认管理员账号登录后查看完整演示数据。
3. 若希望重新获得完整演示效果，建议使用空库启动，让 Flyway 从 `V1` 到 `V10` 完整执行。
