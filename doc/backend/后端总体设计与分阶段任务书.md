# 后端总体设计与分阶段任务书

## 一、文档目的

当前前端页面与交互已经基本满足预期，下一阶段的重点工作转为：

1. 搭建后端基础框架；
2. 建立可持续扩展的数据库结构；
3. 让现有前端页面逐步接入真实后端数据；
4. 采用“先可运行、再逐步完善”的方式，分阶段推进，避免一次性开发过大导致失控。

本文件用于指导后端与数据库建设，后续可拆分为多个任务，逐步交给 Codex 执行。

---

## 二、建设原则

## 2.1 先通后全

优先目标不是一次性把所有业务全部做完，而是先让现有前端页面可以稳定获取真实数据，形成“可运行的前后端闭环”。

优先级顺序建议：

1. 首页综合态势相关接口
2. 地区/地图/情报流/快讯条相关接口
3. 数据源、采集任务、数据包等后台管理接口
4. 用户、权限、日志等系统接口
5. 后续统计分析、规则预警、专题服务扩展

---

## 2.2 先单体模块化，后续再考虑拆分

本项目当前阶段不建议直接做微服务。

建议采用：

- **单体应用**
- **按业务模块分包**
- **统一数据库**
- **统一鉴权**
- **统一异常处理**
- **统一返回结构**

原因：

1. 当前项目处于快速建设阶段，直接微服务会增加复杂度；
2. 当前重点是尽快跑通现有前端页面；
3. 单体模块化更适合 Codex 分步生成与维护；
4. 后期若项目规模变大，可再基于模块边界拆分服务。

---

## 2.3 先围绕前端页面落地，不做脱离页面的空架构

后端建设必须以当前前端已有页面为中心，围绕页面数据需求设计接口和表结构，避免做出一套“理论上很完整但实际前端接不上”的系统。

---

## 2.4 核心思路：以“情报主表”为中心驱动多页面

后端数据设计不应为每个页面单独建一堆割裂表。

建议以 **情报主表** 为核心，再通过地区、事件类型、标签、热度、坐标、来源等字段，支撑以下多个页面和模块：

1. 首页快讯条
2. 首页情报流
3. 首页地图点位
4. 首页风险/热点/重点关注
5. 事件列表页
6. 情报详情页
7. 区域联动统计
8. 热点聚合分析

也就是说：

- **快讯条** = 情报主表中的“高优先级 + 最近时间”的数据
- **情报流** = 情报主表中的“按时间排序”的数据
- **地图点位** = 情报主表中“带坐标 + 带地区”的数据
- **热点模块** = 情报主表按地区/标签/类型进行聚合
- **重点关注** = 情报主表中“重点级别高”的数据

这样可以大幅降低数据库和接口的复杂度。

## 三、技术选型建议

## 3.1 后端技术栈

建议统一采用以下技术栈：

- Java 21
- Spring Boot 3.x
- Maven
- MySQL 8.x
- MyBatis-Plus
- Lombok
- MapStruct
- Validation
- SpringDoc OpenAPI / Knife4j
- Flyway（数据库版本管理）
- Redis（可选，第二阶段接入）
- Sa-Token 或 Spring Security（二选一，建议当前阶段优先简单可控）

---

## 3.2 推荐组合

为了更利于快速落地，建议当前阶段采用：

- **Spring Boot 3 + MyBatis-Plus + MySQL + Flyway + Knife4j**
- 权限先做基础版，不一开始就上复杂 RBAC 深度控制
- Redis 可先预留配置，不要求第一阶段必须用到

---

## 3.3 不建议当前阶段就做的内容

以下内容当前不建议强行上：

1. 微服务拆分
2. 消息队列强依赖
3. 复杂工作流引擎
4. 多数据库并行
5. 大规模 ES 检索集成
6. 复杂规则引擎

这些内容可以在后续第二轮、第三轮迭代时再考虑。

---

## 四、后端总体功能边界

后端第一阶段建议围绕以下几个业务域建设：

### 1. 系统基础域
用于支撑后端最基础运行能力。

包括：
- 用户
- 角色
- 登录
- 基础字典
- 操作日志
- 统一异常
- 统一返回
- 接口文档
- 配置管理

### 2. 地区与地图域
用于支撑海淀区全域与重点街镇联动。

包括：
- 地区树
- 地区边界
- 地图图层
- 地区切换
- 地图点位查询

### 3. 情报与事件域
用于支撑首页快讯、情报流、详情页、风险事件、热点事件等。

包括：
- 情报主数据
- 情报来源
- 事件类型
- 标签
- 情报详情
- 重点级别
- 热度
- 审核状态
- 坐标与地址

### 4. 首页综合态势域
用于支撑首页所有联动接口。

包括：
- 概览统计
- 区域统计
- 快讯条
- 情报流
- 地图点位
- 热点聚合
- 重点关注
- 风险/突发/热点分类

### 5. 数据源与采集任务域
用于支撑“数据源管理”“采集任务管理”等页面。

包括：
- 数据源
- 采集任务
- 任务执行状态
- 数据接入方式
- 启停状态
- 任务日志（后续可补）

### 6. 数据服务与成果交付域
用于支撑专题服务、数据包下载、光盘交付记录等。

包括：
- 数据包
- 数据包文件
- 服务专题
- 发布记录
- 下载记录
- 交付记录

## 六、统一开发规范建议

## 6.1 API 返回结构统一

统一返回格式建议如下：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

分页结构建议：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "records": [],
    "total": 0,
    "pageNum": 1,
    "pageSize": 20
  }
}
```

---

## 6.2 表字段规范

所有核心业务表统一包含以下基础字段：

* `id`：主键
* `created_by`
* `created_time`
* `updated_by`
* `updated_time`
* `deleted`
* `remark`

建议：

* 主键统一使用 `BIGINT`
* 逻辑删除统一使用 `TINYINT`
* 时间统一使用 `DATETIME`
* 字符集统一 `utf8mb4`

---

## 6.3 命名规范

* 表名：`sys_` / `biz_` 前缀
* 实体名：单数
* Controller：`XxxController`
* Service：`XxxService`
* ServiceImpl：`XxxServiceImpl`
* Mapper：`XxxMapper`
* DTO：入参对象
* VO：返回对象

## 十、首页接口设计建议

首页当前是优先级最高的后端接入对象，因此第一轮重点围绕首页出接口。

---

## 10.1 地区切换接口

### 获取地区树

`GET /api/regions/tree`

返回内容：

* 海淀区全域
* 下级重点街镇/街道/片区
* 每个地区的基础信息
* 排序信息

### 获取地区详情

`GET /api/regions/{id}`

返回内容：

* 地区基础信息
* 中心点
* 默认缩放级别
* 是否启用

### 获取地区边界

`GET /api/regions/{id}/geo`

返回内容：

* GeoJSON
* 简化版GeoJSON

---

## 10.2 首页概览接口

### 获取首页总览统计

`GET /api/dashboard/overview`

返回内容：

* 全区总事件数
* 风险事件数
* 突发事件数
* 热点事件数
* 重点关注数
* 当前活跃度
* 最近更新时间

### 获取地区态势摘要

`GET /api/dashboard/region-summary?regionId=xxx`

返回内容：

* 当前区域名称
* 当前区域一句话摘要
* 当前区域各项指标
* 与全区对比占比

---

## 10.3 地图相关接口

### 获取地图图层配置

`GET /api/map/layers`

### 获取地图点位

`GET /api/map/points`

建议参数：

* `regionId`
* `eventType`
* `startTime`
* `endTime`
* `keyword`

返回内容：

* 点位ID
* 标题
* 地区
* 类型
* 坐标
* 热度
* 简要摘要
* 时间

---

## 10.4 快讯条接口

### 获取实时情报快讯

`GET /api/intel/ticker`

建议参数：

* `regionId`
* `eventType`
* `limit`

返回内容：

* 时间
* 地区
* 类型
* 摘要
* 详情ID

---

## 10.5 主情报流接口

### 获取情报流列表

`GET /api/intel/stream`

建议参数：

* `regionId`
* `eventType`
* `pageNum`
* `pageSize`
* `keyword`

返回内容：

* 标题
* 时间
* 地区
* 类型
* 来源
* 摘要
* 热度
* 标签
* 详情ID

### 获取情报详情

`GET /api/intel/{id}`

---

## 10.6 热点与重点关注接口

### 获取热点专题

`GET /api/dashboard/hotspots?regionId=xxx`

### 获取重点关注对象

`GET /api/dashboard/focus-targets?regionId=xxx`

## 十四、当前阶段不建议 Codex 一次性做完的事情

下面这些内容暂时不要一次性交给 Codex 整体生成：

1. 所有模块一起生成
2. 登录权限、数据源、采集任务、首页聚合、文件中心全部一轮做完
3. 数据库一次性设计过于复杂
4. 把未来可能才需要的 AI 分析引擎、规则引擎、ES 检索一起写进第一版
5. 一开始就写复杂大 SQL 和大批量统计任务

原因是：

* 容易超上下文
* 容易生成不稳定
* 后续修复成本大
* 与当前“先让前端跑通”目标不一致

---

## 十五、Codex 执行约束建议

后续发给 Codex 时，应增加以下约束：

1. 每次只做一个 Task，不跨任务扩展。
2. 除当前任务明确要求外，不擅自增加额外框架。
3. 生成代码时必须补充必要注释。
4. 优先保证工程可运行，不优先追求花哨抽象。
5. 生成后同步更新 `/doc` 下对应文档。
6. 数据库变更必须通过 Flyway 脚本，不允许手工漂移。
7. 接口返回结构必须统一。
8. 所有 VO/DTO/Entity 分层清晰。
9. 当前阶段优先围绕前端页面联调需要输出接口。
10. 每完成一个任务，都要给出变更清单、运行方式、验收方式。

## 十六、最终目标

这套后端建设完成后，应达到以下目标：

1. 前端首页综合态势能够接入真实数据；
2. 地区切换、地图点位、快讯条、情报流都具备后端支撑；
3. 数据源、采集任务、数据包管理等后台页面具备基础管理能力；
4. 数据库结构清晰，不混乱、不重复建模；
5. 后端工程具备可持续扩展能力；
6. 后续无论是继续微调前端，还是增加 AI 分析、规则预警、统计分析模块，都有稳定底座。
