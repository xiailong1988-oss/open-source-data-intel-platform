# 项目开发总规则

本文件是当前项目长期有效的总规则。后续任何开发、修复、重构、文档调整，都必须先阅读本文件，再阅读 [当前执行状态.md](E:/codex-test/open-source-data-intel-platform/doc/rules/当前执行状态.md)。

## 1. 执行顺序规则

1. 所有新增开发必须先读 `rules/`，再读 [当前执行状态.md](E:/codex-test/open-source-data-intel-platform/doc/rules/当前执行状态.md)。
2. 所有结构性改动必须先在 `tasks/当前任务/` 记录，再开始执行。
3. 所有任务执行后必须同步更新状态文件和当前任务文档。
4. 后续 Codex 默认以 `rules + 当前执行状态 + 当前任务文档` 作为继续执行的最小上下文。

## 2. 开发原则

1. 优先做结构化、可维护的实现，不允许只做表面补丁。
2. 不允许因为局部问题随意推翻全局结构。
3. 不允许为了修复局部 UI 而造成其他页面错乱。
4. 不允许为了赶进度把逻辑堆进单个页面文件。
5. 不允许只换颜色、不调整布局逻辑就宣称界面重构完成。
6. 不允许只做静态展示，不做必要联动和反馈。

## 3. 界面与布局规则

1. 页面开发必须兼顾高分辨率和常规分辨率。
2. 默认适配范围至少覆盖 2880、1920、1600、1440 四类宽度。
3. 页面必须基于统一布局容器、统一间距、统一面板体系实现。
4. 优先使用 flex / grid / minmax / overflow / breakpoint 解决布局，不允许依赖绝对定位硬压页面。
5. 不允许用大量写死宽高去“适配”单一分辨率。

## 4. 业务稳定性规则

1. 不允许随意破坏已有路由、菜单、筛选、详情抽屉、钻取跳转等核心交互。
2. 任何界面整改都必须以“不破坏原有业务逻辑”为前提。
3. 任何重构后都必须至少自检：构建、菜单切换、详情打开、主要跳转、空态与加载态。

## 5. 文档规则

1. 规则类文档只放在 `doc/rules/`。
2. 任务类文档只放在 `doc/tasks/`。
3. 参考资料与需求原文只放在 `doc/reference/`。
4. 旧版文档必须归档，不允许继续和当前规则、当前任务混放。
5. 每次完成任务后，必须在当前任务文档末尾补“本次改动说明”。

## 6. 注释规则

1. 所有新增函数和核心模块都要写清晰注释。
2. 关键布局文件、关键组件、关键筛选逻辑、关键路由逻辑必须写注释。
3. 复杂 UI 结构和联动逻辑，注释必须说明“为什么这样做”。
4. 命名与注释细则见 [代码注释与命名规范.md](E:/codex-test/open-source-data-intel-platform/doc/rules/代码注释与命名规范.md)。

## 7. 任务闭环规则

每次任务完成后必须同时产出：

- 改动说明
- 涉及文件清单
- 新增组件说明
- 涉及接口说明
- 风险点说明
- 验收结果说明
- 文档更新记录
