# 前端界面与交互规范

本文件用于约束当前项目的页面布局、视觉层级和交互表现，后续所有界面整改与新增页面都必须遵守。

## 1. 布局总原则

1. 统一使用栅格化布局思想，不允许自由堆叠导致重叠。
2. 所有页面必须按需具备清晰的：页头 / 筛选区 / 主体区 / 辅助区 / 底部区。
3. 主内容区必须区分主视图与辅助信息面板，不能所有区块权重相同。
4. 页面应保持呼吸感，不允许把所有模块无差别塞满屏幕。
5. 首屏必须围绕“用户第一眼先看到什么”组织，不允许非关键说明、系统状态和弱摘要抢占主位。

## 2. 高分辨率适配规则

1. 必须兼顾 Chrome 下 2880×1920 分辨率显示效果。
2. 在 2880 宽度下不得出现重叠、截断、按钮不可点、图表被压缩等问题。
3. 同时必须兼顾 1920 / 1600 / 1440 等常规办公宽度。
4. 布局优先通过 `grid`、`flex`、`minmax()`、`clamp()`、合理断点实现。
5. 避免固定高度导致内容挤压，尤其是图表、地图、抽屉、详情区。

## 3. 间距与尺寸规范

1. 页面级主间距使用统一 gap，不允许每页自定义一套密度。
2. 卡片、表格、图表、地图、筛选区、弹窗之间的边距统一使用同一套 spacing token。
3. 按钮、输入框、下拉框、标签的尺寸要在同一视觉节奏上。
4. 图表容器高度必须有统一策略，避免多个图表并排时高低错乱。

## 4. 面板与卡片规范

1. 卡片必须统一圆角、边框、阴影和背景层级。
2. 地图、图表、表格、详情抽屉等都视为“面板”，要遵守同一套面板规范。
3. 页头卡片和信息卡必须有明确层级，不允许所有面板视觉权重一致。
4. 地图和主要分析区优先级高于辅助列表与说明区。
5. 摘要卡、观察卡和 support card 默认必须尽量紧凑，不能只靠大面积留白占位。

## 5. GIS 工作台规范

1. 只要页面被定义为“综合态势”或“态势工作台”，GIS 主视图必须成为第一视觉中心。
2. GIS 主视图周围的信息必须围绕地图组织，不允许地图与若干普通卡片等权并列。
3. 图层切换、热点区域、重点事件、专题建议和侧边情报面板都应服务于地图，不得脱离地图单独堆砌。
4. 地图默认只展示最关键的信息，次级信息通过图层、筛选、右侧面板和底部支撑区承接。
5. 地图加载失败、无点位、无联动对象时必须提供明确空态或重试反馈。
6. 如果首页被定义为“地图主导型驾驶舱”，首屏不得由统计卡片抢占主位，地图必须先于卡片出现。
7. 若驾驶舱明确限定某个区域（例如海淀区），首页地图和首屏说明必须围绕该区域组织，不允许继续泛化到更大的范围。
8. 高密度点位场景下必须默认启用降噪策略，只展开高优先级标注，其余点位用静默符号、分组区块、轮播条带或聚合逻辑承接。
9. 地图中的点位、片区、列表和详情跳转必须形成闭环，不允许只画点不提供继续追踪入口。
10. 次级状态信息如“当前值守 / 最新刷新 / 图层摘要”不得以大片 HUD 覆盖主画布，应降级为徽标、工具状态或侧边承接内容。
11. 重要提醒可以通过高亮、闪烁、脉冲、轮播等方式强调，但不得以遮挡地图作为代价。
12. 首页地图控制区应优先使用紧凑工具按钮、可展开面板、滚动容器等方式收纳，不允许长期用大块控制卡压缩地图画布。
13. 若使用在线底图，必须提供本地或离线可用的回退底图，并明确处理加载失败反馈与归因说明。
14. 地图上的重点预览框必须与点位形成清晰的锚定关系，优先通过 popup、callout 箭头或 leader 线实现，不允许出现“预览框浮在地图上却不知对应哪个点”的情况。
15. 地图主导型首页的骨架以“当前任务文档”的版式映射为准；若当前任务明确采用“地图顶部窄横条 + 右侧情报滚动流”，则不得再强行恢复旧版底部主轮播。
16. 高频控制项必须默认首屏可达；可以做成地图顶部窄横条或轻量展开面板，但不能再堆成大面积左侧模块墙。
17. 情报流或轮播承接层必须与地图、焦点解释区形成三方联动，点击地图点位、点击情报项、切换图层都应同步更新。

## 6. 标题与信息层级规范

1. 页面标题、分区标题、卡片标题必须形成清晰层级。
2. 标题层级应稳定，不允许出现同等级模块却字号和权重混乱。
3. 次级说明文案统一使用较弱层级颜色，避免正文与说明抢权重。
4. 首页与工作台类页面必须明确“主位信息 / 支撑信息 / 继续追踪信息”的三层关系。
5. 驾驶舱类页面的大标题说明默认应克制，能缩成页眉标识的，不应继续占据主内容之前的空间。

## 7. 交互规范

1. 所有可点击区域必须有 hover 态、pointer 和反馈态。
2. 所有页面必须具备空态、加载态、异常态和必要的刷新反馈。
3. 浮层、弹窗、抽屉必须避免越界、内容溢出和按钮压缩。
4. 地图、图表和列表页的联动必须保持一致的跳转与反馈逻辑。
5. 抽屉与弹窗类浮层默认应优先使用 `destroy-on-close`，并在需要时 `append-to-body`，避免残留遮罩干扰全局菜单点击。
6. 任何“第一次能用、后续失效”的交互，都必须补充可重复执行的回归验证路径。
7. 地图信息框必须能清楚说明“发生了什么、在哪发生、什么时候发生、可以继续去哪看”。
8. 从地图进入详情页后，返回上一级时应能继续操作，不得出现菜单失效、页面空白或状态锁死。
9. 驾驶舱类页面必须提供“地图点击”和“列表 / 轮播点击”两种进入重点对象的路径，避免只依赖单一入口。

## 8. 交互回归要求

1. 重点页面变更后，至少要验证：首页、智能检索、智能问答、预警概览、预警记录、报告中心。
2. 验证内容至少覆盖：菜单切换、详情抽屉打开 / 关闭、推荐点击、地图或图表联动、返回列表后的继续操作。
3. 重点工作台页面应优先补到 `scripts/ui-smoke.mjs` 这类可重复执行的回归脚本中，而不是只做手工验证。
4. 如果首页做了 GIS 驾驶舱改造，回归脚本必须补齐图层切换、地图联动、详情跳转、返回首页、侧栏专注模式，以及底图切换 / 回退路径验证。

## 9. 参考风格吸收要求

当前界面整改应重点吸收以下两类参考站点的设计语言：

- [World Monitor](https://www.worldmonitor.app/)
- [Crucix](https://www.crucix.live/)

重点吸收内容：

- 深色控制台式主框架
- 地图 / 主视图优先的布局逻辑
- 模块化信息面板排布方式
- 更克制但更有层级的配色体系
- 高信息密度下仍保持可读性的节奏控制
