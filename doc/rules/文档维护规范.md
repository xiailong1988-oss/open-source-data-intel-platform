# 文档维护规范

本文件用于约束 `doc/` 目录的长期维护方式。

## 1. 目录职责

- `rules/`：长期规则、规范、当前执行状态
- `tasks/`：已执行任务、当前任务、待执行任务
- `reference/`：原始需求、参考资料、归档资料

## 2. 读取顺序

后续任何开发默认按以下顺序读取：

1. [项目开发总规则.md](E:/codex-test/open-source-data-intel-platform/doc/rules/项目开发总规则.md)
2. [当前执行状态.md](E:/codex-test/open-source-data-intel-platform/doc/rules/当前执行状态.md)
3. `tasks/当前任务/` 下的当前任务文档
4. 如需要，再读取 `reference/` 中的原始需求和设计参考

## 3. 更新规则

1. 每次任务执行前，必须先确认当前任务文档是否存在。
2. 每次任务执行后，必须同步更新当前执行状态。
3. 若任务完成，应将任务结果补写到当前任务文档末尾，再归档到 `tasks/已执行任务/`。
4. 旧任务与旧文档必须归档，不允许继续作为当前执行依据混放在根目录。

## 4. 命名规则

1. 规则文件使用明确职责命名，不使用编号堆叠。
2. 任务文件必须体现任务状态：已执行 / 当前 / 待执行。
3. 参考文件必须体现用途：原始需求 / 参考网站 / 风格提炼 / 归档。

## 5. 当前维护结论

从本轮开始，`doc/` 的最小执行入口固定为：

- `doc/rules/项目开发总规则.md`
- `doc/rules/当前执行状态.md`
- `doc/tasks/当前任务/任务-综合态势首页重构为地图主导型情报驾驶舱.md`
