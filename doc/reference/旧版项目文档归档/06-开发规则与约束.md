# 开发规则与约束

本文件是当前项目的长期规则文件。后续所有开发、修复、优化都必须遵守本文件。

## 总体执行前提

- 任何结构性改造前，必须先对照 [11-原始需求说明.md](E:/codex-test/open-source-data-intel-platform/doc/11-原始需求说明.md) 校验产品方向是否偏离。
- 任何按阶段推进的任务，必须先对照 [12-Codex执行任务拆分文档.md](E:/codex-test/open-source-data-intel-platform/doc/12-Codex执行任务拆分文档.md) 确认当前 Phase / Task，不允许跳步开发。
- 如果当前任务属于基础重构阶段，优先补布局、主题、组件、状态等基础能力，不允许借机扩散到无关业务页面。

## 代码层规则

- 不允许随意推翻现有项目结构。
- 不允许在没有计划和说明的情况下做大规模重构。
- 新增页面必须纳入路由配置，并在文档中记录页面作用和入口。
- 新增功能若当前阶段不接后端，必须有独立 mock 数据支撑。
- 新增跳转必须验证可用，不能只写按钮不落地。
- 列表页、详情页、关联对象页尽量形成闭环。
- 新增可点击元素必须有 hover 态、点击反馈和空状态处理。
- 新增页面必须考虑 loading、空状态、错误提示。
- 不允许只做静态展示而不做业务联动，除非需求明确说明只做展示。

## 技术层规则

- 不随意引入收费依赖。
- 不随意引入复杂状态管理。
- 未进入新阶段前，不接真实后端。
- 未进入新阶段前，不接真实模型。
- 未进入新阶段前，不接真实搜索服务或真实 GIS 服务。
- 新依赖必须说明用途、作用范围和替换成本。
- mock 数据优先按模块拆分管理，不允许全堆在单文件。

## 页面与交互规则

- 首页必须保持平台总览入口定位。
- 地图、检索、问答、预警、报告属于重点展示页，不能退化成普通 CRUD 页面。
- 图表如果支持点击联动，就必须保证 query 参数被目标页正确解析。
- 详情页内的关键关联对象必须支持继续跳转。
- 假动作按钮必须给出消息提示，不能点击无反馈。

## 文档层规则

- 新增模块必须同步更新文档。
- 重要页面结构变化必须更新对应 Markdown 文档。
- 新增跳转关系必须更新 [04-页面结构与跳转关系.md](E:/codex-test/open-source-data-intel-platform/doc/04-页面结构与跳转关系.md)。
- 阶段变化和完成情况必须更新 [08-当前开发进度.md](E:/codex-test/open-source-data-intel-platform/doc/08-当前开发进度.md)。
- 重要开发节点和典型问题必须更新 [07-开发过程记录.md](E:/codex-test/open-source-data-intel-platform/doc/07-开发过程记录.md)。
- 涉及通用组件抽离、组件复用策略变化时，必须更新 [13-通用组件说明.md](E:/codex-test/open-source-data-intel-platform/doc/13-通用组件说明.md)。
- 涉及阶段推进时，文档中必须明确本次对应的 Phase / Task 编号。

## 注释规则

- 新增函数必须写注释。
- 新增组件必须写说明性注释。
- 复杂逻辑必须写注释。
- 关键路由逻辑必须写注释。
- 关键初始化逻辑必须写注释。
- query 参数解析逻辑必须写注释。
- 图表数据处理逻辑必须写注释。
- 地图图层切换和地图点击联动逻辑必须写注释。
- 智能问答默认初始化逻辑必须写注释。
- 检索、预警、报告、详情页等关键联动逻辑必须写注释。
- 历史代码不要求一次性全部补齐，但凡新增、修改、重构过的关键代码，必须顺带补注释。

具体写法见 [10-代码注释规范.md](E:/codex-test/open-source-data-intel-platform/doc/10-代码注释规范.md)。

## 结构稳定性约束

- 不允许为了局部功能临时绕开现有布局或路由体系。
- 不允许把大段业务逻辑堆进单个页面而不做最基本的组件拆分。
- 不允许因为一个页面的需要改乱全局样式。
- 不允许删除已有模块文档后不补替代文档。

## Mock 数据规则

- 新增页面前先定义 mock 数据结构，再写页面。
- mock 字段命名必须真实、可读、具备扩展空间。
- 页面筛选和钻取所依赖的 query 参数必须能在 mock 层被正确消费。
- 地图点位、问答结果、预警记录、报告内容等复杂 mock 必须按模块归档。

## 验收与自检规则

每次开发完成后，至少需要自检以下内容：

- 项目是否能正常启动
- `npm run build` 是否通过
- 菜单切换是否正常
- 页面是否存在空白、卡死或点击无反馈
- 新增跳转是否可用
- query 参数跳转后页面是否能正确解析
- 文档是否同步更新
- 注释是否补齐
- 是否符合当前 `Phase / Task` 的范围约束
- 是否仍与原始需求方向保持一致

## 强制执行要求

以下要求不是建议，而是默认强制规则：

- 以后新增函数默认必须写注释。
- 以后新增组件默认必须写说明性注释。
- 以后新增页面若包含复杂联动，必须写初始化说明和跳转说明。
- 以后新增 mock 数据结构，必须在文档中说明用途。
- 以后新增模块，必须同步更新 `doc/` 文档。
- 以后每次阶段性开发完成后，都必须更新 [08-当前开发进度.md](E:/codex-test/open-source-data-intel-platform/doc/08-当前开发进度.md)。
