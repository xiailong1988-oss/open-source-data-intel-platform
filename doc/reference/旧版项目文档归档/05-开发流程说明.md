## 新功能开发的标准步骤

1. 明确需求边界  
   先确认本次只做什么、不做什么，避免跨阶段扩散；如果当前项目已进入任务拆分式改造，必须先确认对应的是 `12-Codex执行任务拆分文档.md` 的哪个 Phase / Task。

2. 补文档或更新文档  
   如果是新模块或较大变化，先更新功能拆解、页面跳转或开发进度文档；如果涉及基础能力和组件抽离，还应同步更新通用组件说明。

3. 设计页面结构  
   先确定放在哪个模块、页面层级是什么、是否复用现有布局。

4. 设计 mock 数据  
   先确定数据结构，再写页面，避免页面字段和 mock 字段不匹配。

5. 写类型定义  
   先补 `src/types`，明确数据结构边界。

6. 写路由和菜单  
   如果新增页面，要先确定路由入口和菜单挂载位置。

7. 写页面与组件  
   优先拆出可复用组件，避免单页过于臃肿。

8. 做跳转与联动  
   页面不是做静态展示，必须考虑“从哪里来、能跳到哪里去”。

9. 自检  
   至少检查：能否运行、有没有空白页、有没有编译报错、点击是否有反馈。

10. 更新文档  
   完成后必须回写开发进度、跳转关系和规则约束相关文档，并记录本次对应的 Phase / Task 执行情况。

## 修 bug 的建议流程

1. 先复现问题
2. 确定问题发生在哪一层  
   页面层、组件层、路由层、mock 层或样式层
3. 只做定向修复，不推翻稳定结构
4. 修复后回归相关联页面
5. 如问题影响联动或初始化逻辑，顺带补注释
6. 如问题涉及产品行为变化，更新对应文档

## 做优化时的建议流程

1. 先说明优化目标  
   是视觉优化、交互优化、结构优化还是性能优化
2. 评估是否会影响现有联动
3. 优先增强，不随意重构
4. 完成后回归关键路径
5. 补文档和注释

## 按任务拆分文档推进时的补充流程

1. 先用 [11-原始需求说明.md](E:/codex-test/open-source-data-intel-platform/doc/11-原始需求说明.md) 校验本次改动是否偏离立项目标。
2. 再用 [12-Codex执行任务拆分文档.md](E:/codex-test/open-source-data-intel-platform/doc/12-Codex执行任务拆分文档.md) 明确本次所属阶段和任务编号。
3. 仅在当前任务范围内改动，避免跨 Phase 提前实现后续内容。
4. 按“改动说明、涉及文件、组件说明、接口说明、风险点、验收结果、文档更新记录”的口径整理交付说明。

## 功能开发顺序建议

后续继续开发或深化时，建议按以下优先顺序推进：

1. 先补关键业务联动和演示闭环
2. 再补数据流与 mock 精度
3. 再补视觉和体验统一
4. 最后再考虑真实接口、真实模型和真实 GIS 接入
