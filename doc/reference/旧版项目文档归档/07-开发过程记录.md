## 典型问题记录

### 路由与菜单问题

曾出现“进入智能问答后再切换菜单异常”的问题，后续通过统一菜单 push 方式、优化 router-view key 和全局 loading 逻辑进行了修复。

### 首页展示项不可点击

早期首页更像静态看板，很多卡片、图表和列表项只有展示没有业务入口。后续通过 query 参数、autoOpen 和目标页解析逻辑补齐了钻取能力。

### 页面联动闭环不足

项目中曾存在“列表能看但详情后无法继续跳转”的问题。后续逐步补齐了内容、实体、事件、预警、报告之间的关联跳转。

### 智能问答首次加载问题

智能问答页曾出现首次点击菜单后内容不显示、刷新后才正常的情况。后续为页面增加了首次挂载兜底初始化逻辑，确保默认问答和历史问题能立即显示。

### 地图模块增强问题

首页最初只有卡片和图表，空间感不足。后续通过 ECharts 注册自定义 SVG 地图，实现了综合、预警、科研、产业四类专题图层和右侧联动面板。

### 基础能力与文档脱节问题

随着页面越来越多，项目曾一度出现“功能能跑，但基础能力没有按统一任务顺序沉淀”的风险。新增原始需求说明和任务拆分文档后，当前已经开始按 Phase 1 的顺序重新收口布局、主题、通用组件和状态机制，避免后续专题页、事件中心和告警中心继续在不稳定底座上叠加。

## 为什么现在要做项目级文档沉淀

随着模块、联动和页面数量增加，仅靠代码上下文已经不够稳定，容易出现以下问题：

- 后续开发时不知道边界
- 修复问题时容易改乱已有结构
- mock 与页面逻辑容易脱节
- 新增模块时缺少统一约束

因此需要把项目定位、需求、架构、页面关系、规则、进度和注释规范系统化沉淀到 `doc/` 中，作为后续开发的统一依据。
