# 技术架构设计

## 当前技术栈

- Vue 3：页面与组件开发主框架
- TypeScript：统一类型约束和可维护性
- Vite：开发启动、构建和打包工具
- Vue Router：菜单、路由和页面跳转管理
- Element Plus：后台布局、表单、表格、抽屉、弹窗等基础 UI
- ECharts：图表和地图演示渲染
- Pinia：当前用于全局布局状态、系统标题、用户信息、顶部工作台上下文和页面刷新状态管理

## 为什么采用这套组合

- Vue 3 + TypeScript 适合构建结构清晰、组件化程度高的中后台系统。
- Vite 启动和构建速度快，适合频繁迭代原型。
- Element Plus 适合后台管理平台，组件成熟，能快速支撑页面完整度。
- ECharts 既能渲染常规图表，也能满足首页地图演示与专题图层切换需求。
- Router + Pinia 已经足够覆盖当前阶段的导航和少量全局状态，不需要引入更重的状态管理。

## 当前前端架构分层

### 页面层

位于 `src/views`，负责模块页面、页面级交互、页面间跳转与页面级筛选状态。

当前典型页面包括：

- 首页综合态势页
- 各模块概览页
- 列表页
- 详情页或详情抽屉
- 智能检索页
- 智能问答页
- 报告详情页

### 组件层

位于 `src/components`，负责复用程度较高的 UI 片段与表现组件。

当前典型组件包括：

- 图表容器组件
- 首页统计卡片
- 地图总览组件
- 首页快捷工作台组件
- 搜索结果卡片
- 问答结果面板
- 页面分区标题组件
- 统一状态组件
- 顶部全局检索 / 问答入口组件

### mock 数据层

位于 `src/mock`，按模块拆分。

当前每个模块都有对应 mock 文件，如：

- `dashboard.ts`
- `dataAccess.ts`
- `dataGovernance.ts`
- `knowledgeBuild.ts`
- `smartSearch.ts`
- `smartQa.ts`
- `analysisWarning.ts`
- `reportCenter.ts`
- `systemManagement.ts`
- `uiConfig.ts`

### 路由层

位于 `src/router`，负责菜单配置、路由结构、模块入口、详情页路径和全局路由 loading 逻辑。

### 状态层

位于 `src/stores`，当前主要用于：

- 侧边栏折叠
- 系统标题
- mock 用户信息
- 顶部专题上下文
- 页面最近刷新时间
- 页面刷新 token

### 样式层

位于 `src/styles`，负责全局变量、全局风格、卡片与表格等通用视觉规则。

当前已经开始沉淀：

- 主题 token
- 风险等级颜色 token
- 全局卡片和可点击态规则
- 统一表单边框和页面间距规则

## 当前目录职责

- `src/layout`：统一后台布局和顶部/侧边导航
- `src/views`：各业务模块页面
- `src/components`：可复用组件
- `src/router`：路由和菜单配置
- `src/mock`：mock 数据
- `src/types`：类型定义
- `src/stores`：Pinia 状态
- `src/styles`：全局样式变量和通用风格
- `src/assets`：静态资源

## 当前为什么使用 mock 数据

当前阶段主要目标是“完成高保真原型和演示闭环”，而不是实现真实数据链路，因此采用 mock 数据有三个直接价值：

- 先验证产品结构是否合理
- 先完成跳转、联动、详情和地图表达
- 为未来真实接口和真实服务保留扩展边界

## 当前模块在架构上的组织方式

### 地图模块

首页地图使用 ECharts 注册自定义 SVG 地图的方式实现，不依赖真实 GIS 服务，但支持：

- 图层切换
- 区域高亮
- 点位渲染
- 热点区域强调
- 右侧联动信息面板
- 跳转到检索、预警、事件、实体、报告

### 图表模块

图表统一通过基础图表组件和图表卡片组件承接，页面负责图表 option 的组织，组件负责渲染和 click 事件透传。

### 顶部全局入口

顶部导航当前通过统一命令栏承接“智能检索 / 智能问答”入口，并带有：

- 专题上下文切换
- 当前页面刷新入口
- 最近刷新时间显示

当前阶段这些能力由本地 `uiConfig.ts` mock 和 `app` store 维护，后续可切换到 `/api/config/ui` 和页面级真实刷新接口。

### 列表页与详情抽屉

大多数列表页采用：

- 表格展示
- 路由 query 解析
- 抽屉/弹窗详情
- 从详情继续跳转到关联对象

### 智能问答页

问答页采用“历史问题区 + 主回答区 + 输入区”的结构。回答面板作为独立组件，页面层负责默认初始化、问答生成和跳转联动。

### 统一状态机制

当前正在逐步从页面内分散的 `el-empty`、loading 和错误提示，收敛到统一状态组件。现阶段已优先在首页工作台辅助区和智能检索结果区接入统一状态组件，为后续事件中心、告警中心和专题页复用做准备。

## 后续如何扩展到真实后端

如果后续接真实后端，推荐做法：

1. 保留页面层和组件层结构不变
2. 在 `src/services` 或 `src/api` 新增接口层
3. 逐步把 `src/mock` 的页面数据替换为接口请求
4. 保留当前 query 参数和页面筛选逻辑，改为驱动真实请求
5. 对列表页引入分页、排序和服务端筛选

## 后续如何扩展到真实模型

如果后续接真实模型：

- 智能问答页保留现有布局和答案结构，只替换答案获取来源
- 智能检索页保留现有筛选和结果结构，把“全文/语义检索”切换接到后端检索服务
- 报告中心可在“新建报告”处调用真实生成任务

## 后续如何扩展到真实检索/地图/GIS 服务

### 检索

- 当前 `smart-search` 页面已经有 query 参数、筛选条件和结果详情结构
- 后续只需替换数据来源与搜索逻辑，不需要推翻页面结构

### 地图 / GIS

- 当前地图的区域、点位、图层、面板结构已经稳定
- 后续可把自定义 SVG 图替换为真实行政区 GeoJSON 或 GIS 服务返回数据
- 当前右侧面板和快捷跳转逻辑可以直接复用

### 预警与报告

- 当前规则、记录、报告详情结构已经清晰
- 后续可逐步替换 mock 数据为真实告警流和报告数据
