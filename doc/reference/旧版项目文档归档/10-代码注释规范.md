# 代码注释规范

本文件定义当前项目以后新增代码、修改代码和关键逻辑补注释时必须遵守的规则。

## 注释目标

注释不是为了重复代码表面含义，而是为了让后续阅读者快速理解：

- 这段代码在做什么
- 为什么这么做
- 它和页面初始化、路由联动、query 参数、mock 数据、图表或地图逻辑有什么关系

## 哪些代码必须注释

以下内容默认必须注释：

- 新增函数
- 新增组件
- 复杂计算逻辑
- 页面初始化逻辑
- 路由跳转逻辑
- query 参数解析逻辑
- 图表数据处理逻辑
- 地图图层切换逻辑
- 地图点位点击和区域联动逻辑
- 智能问答默认初始化逻辑
- 预警、报告、检索、问答等关键联动逻辑
- 影响全局行为的路由守卫和布局逻辑
- 顶部统一命令入口、专题上下文切换和刷新状态逻辑
- 通用状态组件、页面分区头部组件等基础重构期新增能力

## 推荐注释位置

- 函数注释放在函数上方
- 复杂 `computed` / `watch` / `handler` 注释放在声明上方
- 复杂代码段可加块注释
- 必要时可在关键变量行尾加非常短的解释性注释

## 注释风格

- 注释语言统一以中文为主
- 注释优先说明“为什么做”和“做什么”
- 注释要具体，不能写成“设置变量值”这种废话
- 一段复杂逻辑只写一条高质量注释，不要每行都解释

## 历史代码策略

- 不要求一次性为所有旧代码补全全部注释
- 关键模块、后续高频改动模块优先补注释
- 凡是本次修改过的复杂代码，必须顺带补注释
- 以后新增和重构过的代码，默认必须带注释

## 强制规则

- 以后新增函数默认必须写注释。
- 以后新增组件默认必须写组件说明性注释。
- 以后新增页面若包含复杂联动，必须写初始化说明和跳转说明。
- 以后新增 query 参数解析逻辑，必须写注释说明参数用途。
- 以后新增地图、图表、检索、问答、预警、报告等复杂联动逻辑，必须写注释。
- 以后新增基础组件时，必须在组件文件中写用途说明，并同步更新组件说明文档。

## 建议示例

### 函数注释示例

```ts
/**
 * 根据首页卡片类型跳转到目标页面。
 * 这里统一收敛首页钻取入口，避免卡片点击逻辑散落在模板中。
 */
const handleMetricClick = (metricId: string) => {
  metricRouteMap[metricId]?.()
}
```


## 不推荐的注释

以下注释不推荐：

- “给变量赋值”
- “循环数组”
- “点击按钮触发事件”
- 与代码字面完全重复的注释
